## Overview

Call Analysis is the detail view for a single call. It combines recording playback, transcript context, evaluation results, feedback comments, inline coaching, and navigation from snippets or history.

## Real-Call Header Metadata

For real calls, the header displays recipient identity, company, date, duration, language, call type, outcome, and owner assignment when those values exist.

Role behavior:

- `USER`: sees language, call type, and outcome as read-only metadata.
- `TEAM_LEAD`: can update language, call type, and outcome from inline header controls.
- `ADMIN` and `INTERNAL_TOOL_ADMIN`: can update language, call type, outcome, title, and owner assignment.

Language, call type, outcome, and owner changes are optimistic in the frontend and only show toast notifications on errors. Successful saves are silent. Title changes still show success and error toasts.

## Navigation Behavior

Call Analysis preserves breadcrumb context when opened from call collections or hiring flows. If a call cannot be loaded after deletion or missing data, the route redirects admins to team reports and regular users to home.

Source reference: `src/components/analysis/useCallAnalysisNavigation.ts`.

## Recording Playback and Timestamp Navigation

Summary timestamps and play chips seek to the matching moment in the recording and start playback when the current page already has an in-page audio player.

There are two supported paths:

| Page shape | Behavior |
| --- | --- |
| Course call feedback with the sticky summary player | `CallAnalysisFeedback` renders `WaveformPlayer` above the summary. Summary timestamp clicks call `onSummarySeek`, which queues the requested seconds until the waveform player exposes its seek function. Once ready, the player seeks to the timestamp and starts playback without navigating through `commentIndex`. |
| Standard Call Analysis tabs with the conversation player | `AnalysisTabs` switches to the recording tab and passes a pending seek request into `Conversation`. `Conversation` uses its internal `WaveformPlayer` when `bucketUrl` is present, consumes the pending request once seeking is ready, and plays from that timestamp. |

If there is no in-page player for the current path, timestamp navigation falls back to the older URL-driven behavior: the page switches to `tab=recording` and, when a transcript index is available, writes `commentIndex` so the conversation can scroll to the referenced transcript line.

Pending seeks are intentionally request-based, not just a stored timestamp. Each click gets a request id, so repeated clicks on the same timestamp can still be handled. `usePendingSeekRequest` keeps the request until the relevant `WaveformPlayer` reports that metadata is loaded and a seek function is ready, then marks that request handled.

Waveform and compact audio controls stop click propagation from their play, seek, marker, speed, and volume controls. This prevents audio controls inside clickable parent rows or transcript/snippet containers from also triggering the parent row action.

Employee Q&A:

| Question | Answer |
| --- | --- |
| Does clicking a summary timestamp open the transcript line first? | Not when the page has an in-page player. It should seek and play the recording directly. |
| Why might the seek happen after a short delay? | The seek is queued until the audio element has loaded enough metadata for the player to expose a safe seek function. |
| Should support expect `commentIndex` in the URL after a summary timestamp click? | Only on fallback paths without an in-page player. In the player-backed path, the click should not use URL `commentIndex` navigation. |
| What should happen if a user clicks a play button inside a clickable row? | The audio control should handle the click itself. The parent row should not open or navigate because the control stops propagation. |
