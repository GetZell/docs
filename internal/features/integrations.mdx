## tl;dv Integration

The tl;dv settings page lets an admin connect a tenant to tl;dv, tune import filters, review recent meetings, manually import selected meetings into Zell, and optionally enable recurring auto-import.

**Where is it in the app?**

The route is `/settings/tenant/:tenantId/tldv`, built by `appRoutes.toTenantSettingsTldvPage`. The Settings sidebar shows it under `Integrations` with the label `tl;dv`.

Source: `src/appRoutes.ts`, `src/components/layouts/SettingsNav.tsx`

**Who can see it from the frontend?**

The Settings sidebar adds the `Integrations` section only when `session.user.role === UserRole.ADMIN`. Team leads can see the Team settings section, but not the tenant integration links. The tl;dv page component itself does not add an additional role check in the files reviewed; it relies on the app's existing authenticated settings routing and API authorization.

Source: `src/components/layouts/SettingsNav.tsx`, `src/app/(auth)/settings/tenant/[tenantId]/tldv/page.tsx`

**What happens when the tenant is not connected?**

The page shows a `Connect to tl;dv` panel with an API key field and the same import filters used after connection. Clicking `Connect to tl;dv` sends the trimmed API key plus filter payload to `PUT /integrations/tldv`. If the key is empty after trimming, the frontend does not call the API and shows the `Missing API key` toast.

Source: `src/app/(auth)/settings/tenant/[tenantId]/tldv/TldvConnectionPanel.tsx`, `src/app/(auth)/settings/tenant/[tenantId]/tldv/useTldvSettingsViewModel.ts`, `src/hooks/api-hooks/tenants/mutations/useConnectTldv.ts`

**What happens when the tenant is connected?**

The page shows a connected panel, a `Disconnect` action, editable import filters, and a recent meetings section. Meetings are fetched only when `GET /integrations/tldv` returns `connected: true`. Disconnect calls `DELETE /integrations/tldv`.

Source: `src/app/(auth)/settings/tenant/[tenantId]/tldv/TldvSettingsClient.tsx`, `src/hooks/api-hooks/tenants/queries/useGetTldvIntegration.ts`, `src/hooks/api-hooks/tenants/queries/useGetTldvMeetings.ts`, `src/hooks/api-hooks/tenants/mutations/useDisconnectTldvIntegration.ts`

| Frontend action | Hook | Method and endpoint | Request body | Notes |
| --- | --- | --- | --- | --- |

| Import selected meetings | `useImportTldvMeetings` | `POST /integrations/tldv/import-meetings` | `{ meetingIds: string[] }` | Invalidates meetings on success and clears the selection in the view model. |

### Recent Meetings Behavior

Recent meetings are shown in a table with meeting name, happened-at date, organizer name/email, duration, and status. Matching meetings are shown by default. The `Show filtered` switch includes filtered-out meetings in the table; those meetings can still be manually imported unless they are already imported.

Filtered meetings display reason badges for known backend reason codes:

| Backend reason | Frontend label |
| --- | --- |
| `too_short` | `Too short` |
| `title_keyword_missing` | `Keyword missing` |
| `title_excluded` | `Excluded title` |
| `internal_only` | `Internal only` |

Unknown reason strings are displayed raw. `happenedAt: null`, `duration: null`, `organizerName: null`, and `organizerEmail: null` display as `-`. The meetings hook returns `[]` while data is unavailable, and the UI shows `No matching tl;dv meetings found.` when the visible meeting list is empty.

Source: `src/app/(auth)/settings/tenant/[tenantId]/tldv/TldvMeetingsList.tsx`, `src/app/(auth)/settings/tenant/[tenantId]/tldv/TldvMeetingsSection.tsx`, `src/app/(auth)/settings/tenant/[tenantId]/tldv/utils.ts`, `src/hooks/api-hooks/tenants/queries/useGetTldvMeetings.ts`

### Backend Import And Sync

| Concern | Behavior |
| --- | --- |
| API key storage | `PUT /integrations/tldv` validates the key with tl;dv, encrypts it, and stores it under tenant integrations. |
| Meeting list window | `GET /integrations/tldv/meetings` lists recent meetings from the last 30 days, capped at 50. |
| Duplicate detection | Imported state is tenant-scoped and checked against real calls where `metadata.tldvMeetingId` matches. |
| User matching | Manual and automatic imports resolve the organizer to a Zell user by email or name. |
| Recording storage | Imported recordings are saved as `recordings/real-calls/tldv_<meetingId>.mp4` in tenant-private storage. |
| Call metadata | Imported calls store `metadata.audioSource = "tldv"`, `metadata.tldvMeetingId`, title, source URL, and speaker names when available. |
| Transcript handling | Transcript segments are role-assigned before `CallAnalysis` is created. Call and analysis writes happen in one transaction. |
| Queueing | Each successful import enqueues a `call-processing` `process` job. |
| Auto-import | `tldv-sync-meetings` runs every 15 minutes and imports only matching meetings when `autoImportEnabled` is true. |
| Retry behavior | Provider failures bubble up. Scheduled sync advances `lastSyncedAt` only when matching meeting imports did not fail. |

Backend defaults are `minCallDurationSeconds = 300`, `onlyExternal = true`, and `autoImportEnabled = false`. Keyword arrays are trimmed, lowercased, and deduplicated before storage. The request schema rejects whitespace-only API keys, title keywords, exclude keywords, and meeting IDs for manual import before the service tries to connect to tl;dv or import a meeting.

Source: `src/services/integration/tldv.ts`, `src/ports/adapters/tldvAdapter.ts`, `src/jobs/workers.ts`, `src/jobs/handlers/scheduled.ts`, `src/jobs/types.ts`, `src/lib/storage/paths.ts`

### tl;dv Defensive Validation

The backend now treats tl;dv provider meeting fields as required where Zell cannot safely recover. Meeting `id`, `name`, and `happenedAt` must be non-empty strings, and `happenedAt` must parse as a real date string. Transcript entries must include non-empty `speaker` and `text` strings plus finite numeric `startTime` and `endTime` values.

If tl;dv returns malformed meeting or transcript data, Zell throws a provider/integration error instead of creating a call with a fallback title, fallback date, missing speaker, blank transcript text, or invented timing. Manual imports record that meeting as `Import failed`; scheduled sync logs the failed meeting, counts it as skipped, and does not advance `lastSyncedAt` when a matching meeting import fails. The 120-minute scheduled-sync overlap is intentionally unchanged, so the next successful sync can retry the overlapped window.

Imported calls now use the tl;dv meeting `name` and `happenedAt` directly. Zell does not synthesize a replacement title or date for imported tl;dv meetings. After the call and optional analysis are written, the service awaits enqueueing the `call-processing` `process` job before reporting that the import succeeded.

The old local helper script `scripts/uploadTldvRecordings.ts` was removed from the app repo. Employee runbooks should not reference it for tl;dv upload or backfill work.

Source: `src/schemas/integrationSchemas.ts`, `src/ports/adapters/tldvAdapter.ts`, `src/ports/tldvPort.ts`, `src/services/integration/tldv.ts`

### tl;dv Edge Cases for Employee Q&A

| Question | Accurate answer |
| --- | --- |
| Can an admin import a meeting that does not match the filters? | Yes, if `Show filtered` is enabled and the meeting is not already imported. The copy explicitly says filtered meetings can still be imported manually. |
| Can already imported meetings be selected again? | No. Row checkboxes for `alreadyImported` meetings are disabled, and bulk select includes only meetings where `alreadyImported` is false. |
| What does the table header checkbox select? | The header checkbox toggles all currently visible importable meetings. With `Show filtered` off, that means matching visible meetings only. |
| Does refreshing meetings require saving filters first? | No. The `Refresh` button calls the meetings query refetch. Saving filters separately patches settings and invalidates the meetings query. |
| Does the frontend pass the tenant ID in tl;dv endpoint URLs? | No. The hooks use tenant ID in React Query keys, but the endpoint paths are `/integrations/tldv...`; tenant scoping is handled outside the visible URL by the app/API context. |
| What happens after a successful import? | The meetings query is invalidated, the toast reports the imported count returned by the API, and selected meeting IDs are cleared. |
| Can an admin save a tl;dv API key or keyword that is only spaces? | No. The frontend trims the API key before connect, and backend request validation rejects whitespace-only API keys, title keywords, exclude keywords, and manual import meeting IDs. |
| What should support infer from `Import failed` for a tl;dv meeting after this cleanup? | It may indicate malformed provider data, not only a network or storage issue. Check backend logs for `Invalid tldv meeting response` or `Invalid tldv transcript response`; Zell no longer silently imports those calls with fallback names, dates, speakers, or timings. |
| Does auto-import move past a matching meeting that failed because provider data was malformed? | Not as a successful sync. Matching import failures are logged and skipped for that run, and `lastSyncedAt` is not advanced when any matching import fails. |
| Can employees still use `scripts/uploadTldvRecordings.ts` for manual tl;dv uploads? | No. That local one-off script has been deleted and should not be used in support guidance. |
