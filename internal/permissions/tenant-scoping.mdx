# Tenant Scoping

Last verified: 2026-07-09

## Summary

Tenant and user scoping is applied through authentication context, resource loaders, query helpers, and service-layer filters. Most user-facing endpoints scope to the authenticated user or tenant before returning data.

## Authentication Context

`requireAuth` resolves the authenticated user from bearer token or cookie session, then loads the database user. The request context includes:

| Field | Source/meaning |
| --- | --- |
| `userId` | Authenticated user's ID. |
| `tenantId` | Current tenant ID from token/session, validated as UUID. |
| `workspaceId` | Loaded from the database user when available. |
| `managerId` | Loaded from the database user when available. |

## Resource Loader Scoping

`createResourceLoader()` loads resources with:

| User role | Tenant argument passed to loader |
| --- | --- |
| `INTERNAL_TOOL_ADMIN` | `null` |
| Any other role | `user.tenantId` |

This means loaders that support `tenantId = null` can intentionally allow internal admin cross-tenant lookup. Other roles remain tenant-scoped.

## Calls History Scope

Calls History uses the authenticated user's ID:

- Route: `GET /calls/history`
- Handler: `getCallHistory`
- Service call: `CallService.getUserCalls(userId, filters)`
- Query scope: `DB.calls.withUser(userId)`

This means the standard Calls History endpoint returns calls owned by the authenticated user, not all tenant calls.

## User Feedback Inbox Scope

The user Feedback Inbox uses the authenticated user's ID:

- Route: `GET /calls/inbox-snippets`
- Handler: `getInboxReview`
- Service call: `InboxReviewService.getCurrentItem(userId)`
- Pool query: `queryInboxPool(userId, focusSignalIds)`

The inbox pool only includes calls where `Call.userId` matches the current user.

## Manager Pending Replies Scope

Pending replies are tenant-scoped and can be subtree-scoped.

| Step | Behavior |
| --- | --- |
| Route | `GET /admin/feedback/pending-replies` |
| Tenant | Uses `ctx.tenantId`. |
| Team-lead subtree | `teamLeadSubtree(ctx)` returns subtree user IDs only when role is `TEAM_LEAD`; otherwise it returns `null`. |
| Query | `queryPendingManagerReplySnippets(tenantId, userIds)` filters `Workspace.tenantId` and optionally `Call.userId IN userIds`. |
| Grouping | Service groups snippets by user, then call. |

Pending reply rows require:

- At least one reply from a `USER`.
- No later reply from `ADMIN`, `TEAM_LEAD`, or `INTERNAL_TOOL_ADMIN` after the last user reply.

Skipping a pending reply creates an internal manager reply with text `__ZELL_MANAGER_SKIPPED__`, after validating tenant and optional allowed user IDs.

## Employee Answer Pattern

If asked, "Can someone see another user's data?", answer:

Most regular endpoints are scoped to the authenticated user or tenant. Calls History and Feedback Inbox use the current user's ID directly. Admin/manager workflows use tenant scope and may further restrict team leads to their reporting subtree. `INTERNAL_TOOL_ADMIN` is special in resource loaders because it can pass `tenantId = null` where the loader supports cross-tenant lookup.
