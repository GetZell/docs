# Manager Feedback Inbox Filters

Last verified: 2026-07-09

## Summary

The manager Feedback Inbox is the pending-replies inbox shown on the authenticated home page for manager roles. It lists reps who have replied to feedback and still need a manager/admin response.

This is separate from the user Feedback Inbox, which is the daily review inbox for a user's own unread coaching feedback.

## Visible UI Filters

| Filter | Visible to manager | Behavior |
| --- | --- | --- |
| Rep/user selection | Yes | Left column lists users with pending snippets. Selecting a user shows that user's first pending snippet. |
| Manual date range | No | Not exposed in the manager inbox UI. |
| Score filter | No | Not exposed in the manager inbox UI. Score and coachability are displayed on the snippet card. |
| Signal/category filter | No | Not exposed in the manager inbox UI. Signal name can display on the snippet. |
| Agent filter | No | Not exposed as a control. Agent names are displayed under each user row when available. |

## Backend Pending Reply Rules

The backend builds the manager inbox through `queryPendingManagerReplySnippets(tenantId, userIds)`.

| Rule | Current behavior |
| --- | --- |
| Tenant scope | Rows must belong to the current `tenantId` through `Workspace.tenantId`. |
| Team-lead subtree | If the request user is `TEAM_LEAD`, `teamLeadSubtree(ctx)` returns allowed user IDs and the query filters calls to that subtree. |
| User reply required | A snippet is pending only if at least one `USER` reply exists. |
| Manager response missing | A snippet is pending only if there is no later reply from `ADMIN`, `TEAM_LEAD`, or `INTERNAL_TOOL_ADMIN` after the last user reply. |
| Sort order | Rows are ordered by call `startedAt` descending, then `transcriptIndex` ascending with nulls last. |
| Grouping | Service groups snippets by rep/user, then by call. |

## Backend Snippet Selection

For the full backend selection flow, see [Backend snippet selection](../features/backend-snippet-selection).

Short version:

1. `queryPendingManagerReplySnippets()` finds feedback rows with an unresolved user reply.
2. `FeedbackService.getPendingManagerReplies()` hydrates those rows through `buildSnippetsFromRows()`.
3. The service groups hydrated snippets by rep/user and call.
4. The frontend flattens the selected user's snippets and shows the first pending snippet.

## Frontend Freshness

| Action | Query behavior |
| --- | --- |
| Load manager inbox | `useGetManagerPendingReplies` fetches `/admin/feedback/pending-replies` with query key `['manager-pending-replies']`. |
| Add reply | `useAddFeedbackReply` invalidates `['manager-pending-replies']`, so replied snippets should leave the pending manager inbox after refresh. |
| Skip pending reply | `useSkipManagerPendingReply` optimistically removes the feedback from `['manager-pending-replies']`, then invalidates the query. |

## Skip Behavior

Skipping does not delete the feedback. Backend validates the feedback is tenant-scoped and, for subtree-limited managers, belongs to an allowed user. It then creates an internal manager reply with text `__ZELL_MANAGER_SKIPPED__`.

That reply text is an internal state marker. Frontend reply surfaces filter it out through `getVisibleFeedbackReplies()`, so it should not appear in inline comment threads, snippet reply counts, collapsed last-reply previews, or snippet reply panels.

## Role Visibility

The frontend renders `ManagerInbox` when the session role is `ADMIN` or `TEAM_LEAD`.

Open verification: backend `adminFeedbackRouter` has router-level `requireAdmin`, while pending-replies routes also use `requireTeamLeadOrAdmin`. If team leads are intended to use this inbox, verify the route mounting or middleware order because router-level `requireAdmin` may make the route admin-role-only in practice.

## Employee Answer Pattern

If asked, "Why does a manager see this item?", answer:

The manager inbox shows feedback where a regular user has replied, and no manager/admin has replied after that latest user reply. The inbox is tenant-scoped and may be limited to a team lead's reporting subtree.
